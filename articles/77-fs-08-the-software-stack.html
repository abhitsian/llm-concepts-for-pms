The Software Stack

Module 8 of From Sand to Software
┈┈┈┈

Between the raw hardware and the AI models you call from your product, there are layers of software that most people never think about. These layers — compilers, serving frameworks, optimization libraries — are where enormous competitive advantage lives. They're invisible to end users but determine your product's cost structure entirely.

CUDA: The Real Moat

Nvidia doesn't just sell a chip. They sell a full-stack solution where the compiler, the libraries, and the hardware are perfectly co-designed. CUDA is the programming interface, 20 years deep, that every meaningful piece of AI software is built against. Every PyTorch op, every transformer layer, every quantization technique — tuned for CUDA.

AMD has great hardware. Intel has Gaudi. But Patel puts it bluntly: "AMD has great hardware. Nvidia wins because CUDA took 15 years to build. You can't clone a software ecosystem." Switching to AMD means rewriting or losing performance on millions of lines of code. The software stack is stickier than the hardware.

vLLM: The Most Important Software You've Never Heard Of

Developed at UC Berkeley, vLLM figured out how to manage GPU memory the way operating systems manage computer memory — dynamically, efficiently, with almost no waste. The key innovation: PagedAttention, which stores the KV cache (the model's working memory) in non-contiguous memory pages instead of requiring one large contiguous block.

Before vLLM, up to 80% of GPU memory was wasted during inference. Fragmented allocations, pre-reserved blocks sitting empty. After vLLM, waste dropped to under 4%. The practical result: Stripe cut inference costs 73% by migrating to vLLM. Not by buying better hardware. Not by switching models. By using better serving software.

Continuous batching — vLLM's other innovation — keeps injecting new requests as old ones complete, achieving up to 23x throughput improvement over static batching. A single GPU goes from serving 5% of its capacity to serving 80%+. The difference is pure margin.

Quantization: Trading Precision for Speed

Neural network parameters are stored as numbers with a certain precision. Training uses FP32 (32-bit) or FP16 (16-bit). But inference can often use much lower precision — FP8, INT8, even INT4 — with minimal quality loss.

Running in FP8 instead of FP32 can roughly 4x throughput: 4 times as many numbers fit in the same memory, and compute is proportionally faster. Advanced labs use custom CUDA kernels to push INT4 precision. The 280x inference cost reduction over the past 3 years is substantially driven by quantization improvements. Hardware improved ~10x. The rest was software.

Speculative Decoding

Language models generate text one token at a time, sequentially. Each token requires a full forward pass through the model. Speculative decoding breaks this: a small, fast "draft" model predicts the next 5–8 tokens. The main model verifies them all in one parallel pass. When predictions match — and for predictable sequences like common phrases, they usually do — you get multiple tokens for the cost of one forward pass.

2–3x latency reduction. Zero change in output quality. Now standard in vLLM, TensorRT-LLM, and SGLang. Nvidia's H200 showed 3.6x throughput improvement with this technique. When a new model version feels "faster" — speculative decoding is often the story.

The Pattern That Matters

Across the entire stack: the physical layer sets the ceiling, but software determines how close you get to it. A well-optimized software stack on last-generation hardware routinely outperforms a poorly-optimized stack on current-generation hardware. This is why inference cost fell 280x while hardware improved only ~10x. The gap was closed by software.

For your product: don't just ask "which model?" Ask "which serving framework, which quantization, which batching strategy?" Those answers determine whether your AI features have positive unit economics.
