The Software Stack

Module 8 of From Sand to Software
┈┈┈┈

Between the raw hardware and the AI models you call from your product, there are layers of software that most people never think about. These layers — compilers, libraries, serving frameworks — are where enormous competitive advantage lives. They're invisible to end users but determinative of cost, speed, and what's possible.

CUDA: The 20-Year Moat

The reason Nvidia dominates AI isn't the GPU. It's CUDA. CUDA (Compute Unified Device Architecture) is the programming interface that sits between software code and Nvidia hardware. It launched in 2006. Every AI framework — PyTorch, TensorFlow, JAX — is built on top of CUDA. Every optimization technique, every custom kernel, every production deployment system assumes CUDA underneath.

AMD has built hardware with comparable specs. Intel has Gaudi. Google has its own compiler stack for TPUs. But none of them have 20 years of ecosystem. The graduate student who wrote a breakthrough paper used CUDA. The startup that deployed a novel architecture used CUDA. The enterprise ML team that optimized inference used CUDA. Switching means rewriting or losing performance on millions of lines of code.

Hardware can be matched in 3–4 years. Software ecosystems cannot. This is the deepest moat in the AI industry.

The Serving Layer: Where Cost Becomes Margin

Once a model is trained, serving it to millions of users requires a separate software stack optimized for inference. This is where the difference between 30% and 80% GPU utilization lives — and that difference is pure margin.

vLLM is the most important piece of AI infrastructure software most people have never heard of. Developed at UC Berkeley, vLLM figured out how to manage GPU memory the way operating systems manage computer memory — dynamically, efficiently, with almost no waste. The key innovation: PagedAttention, which stores the KV cache (the model's working memory during a conversation) in non-contiguous memory pages instead of requiring one large contiguous block.

Before vLLM, up to 80% of GPU memory was wasted during inference due to fragmentation and pre-allocation. After vLLM, waste dropped to under 4%. The practical result: Stripe cut inference costs 73% by migrating to vLLM. Not by buying better hardware. Not by using a different model. By using better serving software.

Quantization: Trading Precision for Speed

Neural network parameters are stored as numbers with a certain level of precision. Training typically uses FP32 (32-bit floating point) or FP16 (16-bit). But inference can often use much lower precision — FP8, INT8, even INT4 — with minimal quality loss. This is called quantization.

Running a model in FP8 instead of FP32 can roughly 4x throughput: you fit 4 times as many numbers in the same memory, and the compute is proportionally faster. Advanced labs use custom CUDA kernels to squeeze FP4 precision from their hardware. The inference cost reduction over the past 3 years — roughly 280x — is substantially driven by quantization improvements.

Speculative Decoding

Language models generate text one token at a time, sequentially. Each token requires a full forward pass through the model. Speculative decoding breaks this bottleneck: a small, fast "draft" model predicts the next 5–8 tokens, and the main model verifies them all in a single parallel pass. When predictions match (which they often do for predictable sequences), you get multiple tokens for the cost of one forward pass.

This technique delivers 2–3x latency reduction with zero change in output quality. It's now standard in production serving frameworks — vLLM, TensorRT-LLM, and SGLang all support it. When a new model version feels "faster," speculative decoding is often part of the story.

The Pattern

Across the entire software stack, the pattern repeats: the physical layer sets the ceiling, but software determines how close you get to it. A well-optimized software stack on last-generation hardware routinely outperforms a poorly-optimized stack on current-generation hardware. This is why Nvidia's real advantage is CUDA, not the GPU. Why Google's TPU advantage is the co-designed software, not the chip. Why inference cost fell 280x while hardware improved maybe 10x — the rest was software.

First Principles Takeaway

The software between hardware and models is the highest-leverage layer in the AI stack. It's invisible to users but determines your product's cost structure. When you evaluate AI infrastructure, don't just ask "which chip?" — ask "which serving framework, which quantization, which batching strategy?" The answers to those questions determine whether your AI feature has positive unit economics or not.
