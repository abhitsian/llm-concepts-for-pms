How an LLM Actually Runs on Silicon

Inside the AI Machine — Foundations. Primer 4 of 5. Where the hardware story and the model story finally meet.
┈┈┈┈

What this primer gives you: the bridge. Primers 1–3 were hardware. This one connects that hardware to the AI running on it. After this you will understand what a token really is, the two phases of answering a request and why one is compute-bound and the other memory-bound, what the KV cache is and why it eats memory, and the handful of architecture words — attention, FFN, MoE — that the issues assume you know. This is the primer that turns hardware facts into AI understanding.

Tokens: the unit of everything

An LLM does not read letters or words. It reads tokens — chunks of text, each roughly four characters. "Understanding" might be one token; "tokenization" might be three. A page of text is perhaps 500 tokens.

Tokens are the unit of everything in this industry. The model reads tokens in and writes tokens out. You are billed per token in (input) and per token out (output). When the articles say "cost per million tokens" or "tokens per second," tokens are the currency, the speedometer, and the workload all at once. If Primer 1’s atom was the transistor, Primer 4’s atom is the token.

The model is just numbers in memory

From Primer 2: a model is billions of parameters — also called weights — which are simply numbers. A "1.6 trillion parameter" model is 1.6 trillion numbers. Those numbers, stored in memory, are the model. They encode everything it learned.

This is why model size and memory are the same conversation. A bigger model is more numbers, which need more memory to hold and more memory-bandwidth to read. Primer 2’s capacity-vs-bandwidth tradeoff and Primer 3’s precision lever both exist because of this: the model is numbers, and numbers live in memory.

Training vs inference: two completely different jobs

There are two things you do with a model, and conflating them is a common mistake.

Training is building the model — starting from random numbers and adjusting all the parameters, over and over, across an enormous body of text, until the numbers encode useful capability. Training is done once per model. It is staggeringly compute-heavy and costs tens to hundreds of millions of dollars. The first and largest stage is pretraining; later, smaller stages (often using reinforcement learning) refine it. A brand-new "pretrain" is a major event for an AI lab — the coding issue treats OpenAI’s "Spud" pretrain as headline news.

Inference is using the finished model — running it to answer a request. Inference happens billions of times a day. It is where ongoing cost lives, where revenue is earned, and where almost all the hardware drama in these issues plays out. When you hear "serving tokens," that is inference.

Rough analogy: training is writing the textbook; inference is a student using it to answer a question. The writing is a huge one-time effort; the using happens constantly. Most of this series — Cerebras, Groq, the value-capture economics — is about inference.

The two phases of inference: prefill and decode

Here is the most important mechanism in the primer, and the one that makes the Cerebras and Nvidia articles legible. Answering a single request has two distinct phases, and they behave so differently that the industry now runs them on different chips.

Phase 1 — prefill. The model reads your entire prompt. Everything you sent — the question, the pasted document, the system instructions, the conversation history — is processed in one big parallel gulp. Because it’s one large batch of math done all at once, prefill has high arithmetic intensity (Primer 3): lots of math per byte of memory touched. Prefill is compute-bound. It suits GPUs, which have abundant FLOPS. Prefill determines your "time to first token" — the wait before the answer starts.

Phase 2 — decode. Now the model writes the answer — but it can only produce one token at a time. To write each token, it must read the model’s entire set of parameters out of memory, do a modest amount of math, produce one token, and repeat. A 500-token answer is 500 repetitions of "read all the weights, do a little math." That is low arithmetic intensity: lots of memory traffic, little math per byte. Decode is memory-bound. Its speed is set by memory bandwidth, not FLOPS.

Read those two paragraphs again, because everything clicks here:

  • Decode being memory-bound is exactly why SRAM machines (Cerebras, Groq) are fast — their enormous memory bandwidth tears through the memory-bound decode phase.
  • It is why the industry now does disaggregation: run compute-bound prefill on GPUs, run memory-bound decode on a bandwidth-optimized chip. The Nvidia GTC article’s "Attention–FFN Disaggregation" and the AWS–Cerebras "prefill on Trainium, decode on the wafer" deal are both this idea.
  • It is why "fast tokens" is a hardware story: token speed = decode speed = memory bandwidth.

The KV cache: the conversation’s memory

During decode, the model writes token 1, then token 2, and so on. To write token 200, it needs the context of tokens 1 through 199. It would be insanely wasteful to re-read and re-process all 199 every single time.

So the model keeps a running scratchpad of everything processed so far. That scratchpad is the KV cache ("key-value cache" — the name doesn’t matter). It is the conversation’s working memory.

Two facts about the KV cache drive a lot of the issues:

  • It grows. The longer the conversation — the longer the prompt, the longer the answer, the more turns — the bigger the KV cache. A long-context agentic session can have a KV cache of many gigabytes.
  • It lives in memory, competing with the model. The chip’s memory has to hold both the model’s parameters and every active user’s KV cache. This is the squeeze. It is why Cerebras’s tiny 44GB struggles with long context — weights eat most of it, leaving little for KV cache. It is why DeepSeek V4’s "90% smaller KV cache" was a big deal. It is why "context memory storage" is becoming its own product category.

Related term: the context window is the maximum amount of text (in tokens) a model can hold in mind at once — its working-memory ceiling. A "1M-token context window" can hold a long book. Bigger context is more useful but costs more KV-cache memory.

Inside a model: attention and FFN

You don’t need the deep math, but two words recur and you should know what they point at. Every modern LLM is a transformer, built from a stack of identical layers. Each layer has two parts:

Attention. The part that lets each token look at all the other tokens and decide what’s relevant — how a word relates to the rest of the sentence and document. Crucially, attention is the part that reads the KV cache. So attention is memory-heavy: its performance is bounded by loading that cache.

FFN — Feed-Forward Network. The part that processes each token on its own, applying the bulk of the model’s learned knowledge. The FFN is stateless (it doesn’t depend on other tokens or the cache) and compute-heavy.

Two parts, two personalities: attention is stateful and memory-bound, FFN is stateless and compute-heavy. That difference is the entire basis of the Nvidia GTC article’s "Attention–FFN Disaggregation" — put memory-bound attention on GPUs with lots of memory, put compute-bound FFN on a deterministic chip like an LPU. You now know why that split is even possible.

MoE: why models got "sparse"

One more architecture word, because it’s everywhere: MoE, Mixture of Experts.

An older "dense" model used every one of its parameters for every token — expensive. An MoE model is split into many specialist sub-networks called experts, and each token is routed to only a few of them. So the model can have a huge total parameter count (lots of knowledge) while only activating a small fraction for any given token (modest cost to run).

This is why the articles always cite two numbers, like "DeepSeek V4: 1.6T total / 49B active." Total is how big the model is; active is how much it actually uses per token. MoE is how models keep getting bigger without getting proportionally more expensive — and it’s why each expert is small (hundreds of megabytes), a fact the Nvidia disaggregation story relies on.

Serving many users: throughput vs interactivity

A real inference service handles many users at once. The chip processes them in a batch — the batch size is how many requests it serves simultaneously. This creates the central tradeoff of inference economics:

  • Throughput — total tokens per second across all users on the chip. Big batches raise throughput.
  • Interactivity — tokens per second delivered to one user. Big batches lower interactivity, because the chip’s attention is divided.

It is the bus-versus-Ferrari tradeoff: serve many people slowly (the bus — high throughput) or one person fast (the Ferrari — high interactivity). You move along this curve mostly by changing batch size. Every "fast tier," "priority tier," and SRAM-machine pitch in this series is a position on this one curve. (Issue 1, Cerebras, is built around it.)

A few more words you’ll meet

  • Speculative decoding — a speed trick: a small fast model guesses the next few tokens, the big model verifies them in a single pass. Right guesses mean more tokens per step.
  • Distillation — training a small model to imitate a big one, yielding a cheaper model with much of the quality. (How GPT-5.3-Codex-Spark was made.)
  • Reasoning model — a model that generates hidden "thinking" tokens (chain-of-thought) before answering. Far better on hard problems, but burns 5–100x more tokens.
  • Harness — the software wrapped around the model: context management, tools, memory, sandboxing. Claude Code is a harness. Issue 4 argues the harness matters as much as the model.
  • Agent — an AI that runs a multi-step loop on its own (read, plan, act, verify) instead of answering one prompt. The subject of Issue 7.

Vocabulary check

  • Token — ~4 characters; the unit of input, output, and billing.
  • Parameter / weights — the billions of numbers that are the model.
  • Training vs inference — building the model (once, costly) vs using it (constantly).
  • Prefill — reading the prompt; compute-bound; suits GPUs.
  • Decode — writing the answer token by token; memory-bound; suits high-bandwidth chips.
  • KV cache — the conversation’s running memory; grows with length; competes with the model for memory.
  • Context window — the model’s working-memory ceiling, in tokens.
  • Attention / FFN — the two halves of a transformer layer; memory-heavy vs compute-heavy.
  • MoE — many experts, few used per token; total vs active parameters.
  • Throughput vs interactivity — total tokens/sec vs per-user tokens/sec; the bus-vs-Ferrari curve.
