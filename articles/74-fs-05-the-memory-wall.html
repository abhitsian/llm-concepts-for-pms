The Memory Wall

Module 5 of From Sand to Software
┈┈┈┈

Here's something that trips up almost everyone who looks at AI hardware for the first time: the chip's compute power is often not the bottleneck. Memory bandwidth is.

The Chef Analogy

Imagine a chef who can cook 1,000 dishes per minute. Superhuman speed. But the kitchen has a small window through which ingredients are passed, one at a time. It doesn't matter how fast the chef works — they're limited by how fast ingredients arrive. Most of the time, the chef stands idle, waiting.

This is the memory wall. AI chips can compute faster than they can receive data. For every operation during inference, the chip needs to load model parameters — the billions of numbers encoding everything the model knows — from memory into compute. The speed at which this data moves is called memory bandwidth, and it's the binding constraint on AI inference performance today.

HBM: Stacked on Top of the Chip

The memory that feeds AI chips is called HBM — High Bandwidth Memory. It is physically stacked on top of the processor in a single package, connected by thousands of tiny copper pillars called microbumps. The stacking is literal: layers of memory chips bonded on top of each other, then bonded to the logic chip below. This minimizes the distance data travels, maximizing bandwidth.

The H100 has 80GB of HBM3 at 3.35 TB/s. The B200 has 192GB of HBM3e at 8 TB/s. The next generation will have more. But it's still not enough. A large language model has hundreds of billions of parameters. During inference, generating each token requires moving a significant fraction of these parameters through memory. Do the math: if generating one token requires loading 100GB of data, and your bandwidth is 3.35 TB/s, the theoretical maximum is ~33 tokens per second. In practice, it's slower. This is why AI responses feel like they type themselves out gradually — the model is literally waiting for data to arrive from memory, token by token.

The Supply Crunch

Three companies make HBM. SK Hynix (South Korea), Samsung (South Korea), Micron (United States). That's the entire global supply. SK Hynix leads in quality and volume, having bet early and aggressively. Samsung has been catching up. Micron supplies a smaller share.

There is a massive HBM supply crunch. Every new AI chip generation needs more: H100 uses 80GB, B200 uses 192GB, the GB200 NVL72 rack uses over 13 terabytes. Demand is growing faster than HBM manufacturing can expand. This is Dylan Patel's constraint #2: memory. When AI services get rate-limited or slowed, it's often not that the models aren't smart enough — it's that there aren't enough memory chips to run them at full speed.

Why This Changes Your Product Decisions

Longer context windows are dramatically more expensive. When you extend a model's context from 4K to 200K tokens, the intermediate computations (the KV cache — the model's working memory for the conversation) grow proportionally. More context = more GPU memory consumed = fewer users fitting on the same hardware simultaneously. A 200K context query costs roughly 100x more than a 2K context query. Not linearly. Dramatically.

Model size is constrained by memory, not compute. A 1-trillion-parameter model at FP16 requires 2 terabytes just for the weights. No single chip has that. You split across chips, which introduces communication overhead. Memory capacity sets the practical ceiling on model size per chip.

Batch size determines your unit economics. If you can fit more users' queries into GPU memory simultaneously, cost per query drops. Memory capacity directly determines how many concurrent users you serve efficiently. More HBM = bigger batches = lower cost per token = better margins.

Every product decision you make about context length, conversation history, and system prompt size is, at root, a decision about how much HBM you're consuming per user. The PM who understands this builds features with viable economics. The one who doesn't ships features that become more expensive as they succeed.
