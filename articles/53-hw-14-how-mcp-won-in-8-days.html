How MCP Won in 8 Days

Part 14 of the How It Actually Works series
╌╌╌╌

A developer wants her AI coding assistant to read from her company's Jira board, query the production database, & post updates to Slack — all within a single conversation. Six months ago, this meant writing three separate tool integrations: custom JSON schemas for each API, bespoke parsing logic, hand-rolled authentication handling. Now she installs three MCP servers, points her client at them, & it just works. Model Context Protocol did for AI tool connections what USB did for peripherals — replaced a mess of proprietary connectors with one universal standard.

The Spec That Won by Being Boring

MCP's technical spec is almost aggressively simple. It's JSON-RPC — the same remote procedure call format that's been around since 2005 — transported over stdio (for local tools) or HTTP with server-sent events (for remote ones). A server exposes a list of tools, each described by a name, a description, & a JSON Schema for its input parameters. A client discovers these tools, presents them to the model, & routes tool calls to the appropriate server. That's the entire protocol. No custom binary format, no novel serialization, no authentication framework baked in.

This simplicity was the strategy. Anthropic released MCP as open source in late 2024, & within days developers were building servers for everything — GitHub, Postgres, Notion, file systems, web browsers, Kubernetes clusters. The barrier to entry was absurdly low: if you could write a function that takes JSON input & returns JSON output, you could build an MCP server. The initial burst of community adoption was so fast that "8 days" became shorthand for the protocol's breakout moment, though reaching full critical mass took closer to two months.

Why Alternatives Didn't Win

MCP wasn't the first attempt at standardizing tool connections. OpenAI's function calling — introduced in mid-2023 — let models invoke tools, but it was proprietary to the OpenAI API. If you wanted the same tools to work with Claude or Gemini or a local model, you rewrote the integration. LangChain's tool abstraction was framework-specific: powerful if you were already in the LangChain ecosystem, useless if you weren't. Various other projects tried to create universal tool registries, but most were too ambitious — trying to standardize tool discovery, authentication, billing, & versioning all at once.

MCP won by doing less. It standardized exactly one thing: the interface between a model client & a tool server. Authentication? Handle it yourself. Billing? Not MCP's problem. Tool discovery beyond the local server? Out of scope. This minimalism made adoption frictionless cuz developers didn't have to buy into an entire philosophy — they just had to implement a thin interface. The protocol was boring on purpose, & boring is what scales.

Network Effects Kicked In

The flywheel was textbook. Once major clients — Claude Desktop, Cursor, Windsurf, various IDE extensions — supported MCP, building a server became immediately valuable cuz it worked everywhere. Once thousands of servers existed, any new client had a strong incentive to support MCP cuz it gave instant access to a huge ecosystem of tools. Server count drove client adoption; client adoption drove server count. Within months, MCP had the kind of ecosystem density that usually takes years to build.

The ecosystem also benefited from a quality gradient. Simple servers — "read this file," "query this database" — were trivially easy to build & immediately useful. Complex servers — "manage a Kubernetes deployment," "orchestrate a CI/CD pipeline" — came later as the community matured. This meant the protocol delivered value on day one for basic use cases while growing into sophisticated ones organically.

For PMs, MCP's victory carries a broader lesson about standards in fast-moving markets. The winning standard wasn't the most technically sophisticated — it was the one that solved one specific pain point with the least friction. If you're building in a space where a standard is emerging, the strategic question isn't "is this technically perfect?" It's "is adoption happening fast enough to create lock-in?" With MCP, the answer was yes before most teams finished evaluating it. Standards that win by being boring & obvious don't give you time to deliberate. They give you time to adopt or get left behind.
