How Compaction Works

Part 3 of the How It Actually Works series
╌╌╌╌

A developer is three hours into a coding session with Claude Code. They've discussed architecture decisions, debugged a race condition, refactored two modules, & written tests. The conversation is massive — tens of thousands of tokens of back-&-forth. Then, mid-task, the agent pauses. It's hitting the context window ceiling. Without intervention, the session dies. Instead, the system kicks off compaction: it reads the entire conversation, generates a compressed summary, & swaps out the original messages for that summary. The session continues. The developer barely notices.

The Context Window Wall

Every LLM has a finite context window — the maximum number of tokens it can process in a single call. Even with 200K-token windows, long-running agent sessions fill up fast. Each user message, each model response, each tool call & its output, each file read — they all accumulate. A coding agent that reads files, writes code, runs tests, & discusses results can burn through 200K tokens in under an hour of active use.

When the context fills up, you have two options: end the session or make room. Compaction is how you make room. The system takes the full conversation history, feeds it to the model with an instruction like "summarize this conversation, preserving all decisions made, files modified, current task state, & open questions," & gets back a condensed version — typically 10-20% of the original length. This summary replaces the raw conversation history, freeing up space for the session to continue.

The mechanics vary by implementation. Some systems compact the entire history at once. Others use a sliding window — keeping the most recent N messages in full detail while summarizing everything older. Some compact proactively when context usage crosses a threshold (say 80%), while others wait until they're about to hit the wall. Claude Code, for instance, triggers compaction automatically & lets users trigger it manually with a slash command.

The Lossy Compression Problem

Compaction is lossy. There's no way around this. A 50,000-token conversation compressed to 5,000 tokens will lose information. The question is which information gets lost & whether it matters.

Good compaction preserves structural decisions — "we agreed to use PostgreSQL, not MongoDB" — while discarding the deliberation that led there. It keeps track of what files were modified & how. It retains the current task & its subtasks. It drops the back-&-forth exploration, the dead ends, the "actually wait, let me reconsider" tangents that consumed tokens but don't affect what happens next.

Bad compaction loses critical details. It might forget that a specific file was already modified, leading the agent to overwrite its own work. It might lose a constraint the user mentioned early in the session — "don't touch the auth module" — that wasn't restated recently. It might flatten nuanced decisions into oversimplified summaries that cause the agent to make wrong assumptions going forward. When an agent starts behaving erratically after a long session, compaction quality is often the culprit.

The Reliability Ceiling

This is why compaction directly affects product reliability. An agent that compacts well can maintain coherent sessions for hours, handling complex multi-step tasks that span dozens of files & hundreds of decisions. An agent that compacts poorly degrades — slowly at first, then suddenly — as accumulated information loss compounds into confused behavior.

The design challenge for PMs is that compaction failures are subtle. The agent doesn't crash or throw an error. It just starts making slightly wrong decisions cuz it lost context. Users experience this as the AI "forgetting" things they discussed earlier, which erodes trust in a way that's hard to diagnose & harder to fix.

Products that depend on long-running sessions — coding agents, research assistants, workflow automation — need to treat compaction as a first-class concern, not a background utility. The quality of your summarization logic sets the ceiling on how long your agent can work reliably. Ppl building these systems are effectively designing their product's memory, & memory quality determines how much users can trust the product with real work.
