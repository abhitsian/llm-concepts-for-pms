Inference

Module 10 of From Sand to Software
┈┈┈┈

Training creates the model. Inference is using it. Every time you send a message to Claude or ChatGPT, you are running inference — the model takes your input, processes it through billions of parameters, and generates a response one token at a time. Training happens once. Inference happens billions of times per day, across every AI product in the world, and it never stops.

What Actually Happens in 500 Milliseconds

When a user types a message and hits send, two things happen in sequence:

Phase 1: Prefill. The model reads the entire input — system prompt, conversation history, user message — and builds a compressed representation. This is compute-heavy. For a 10,000-token context, prefill alone takes 200–400ms. The model processes all input tokens in parallel, building the KV cache (a stored representation of everything it's read so far).

Phase 2: Decode. The model generates the response, one token at a time. Each token requires a full forward pass through billions of parameters. This is sequential and memory-bandwidth-bound — the chip is mostly waiting for data to arrive from HBM. A 300-token response takes 1–2 seconds.

The KV cache — the model's working memory for the current conversation — grows with context length. Longer conversations consume more GPU memory, which means fewer concurrent users fit on the same hardware. This is why longer context windows cost dramatically more: not linearly, but exponentially in terms of resource consumption.

The Five Levers

Anthropic, OpenAI, and every serious lab has teams of 100+ engineers optimizing these five dimensions of inference:

1. Batching — Process many users' queries simultaneously instead of one at a time. A single query uses ~5% of a GPU's capacity. Continuous batching (vLLM's innovation) keeps injecting new requests as old ones complete, achieving up to 23x throughput improvement. Low-traffic features are disproportionately expensive because batches don't fill.

2. Quantization — Run the model at lower numerical precision. FP8 instead of FP32 can ~4x throughput with minimal quality loss. Advanced labs use INT4 with custom kernels.

3. KV Cache Management — Cache repeated content (system prompts, common prefixes) instead of recomputing for every query. Prompt caching can eliminate the most expensive part of prefill for repeated patterns.

4. Speculative Decoding — Use a small fast model to draft tokens, then verify in bulk with the main model. 2–3x latency reduction, zero quality loss.

5. Model Routing — Route easy queries to small cheap models, hard queries to frontier models. 80% of queries may not need the most powerful model. Intelligent routing captures cost savings without sacrificing quality where it matters.

The Cost-Capability-Speed Triangle

For every AI feature, you can optimize for two of three dimensions, never all three:

Capability (frontier model, full context) vs. Cost (token price, GPU hours) vs. Speed (time to first token, tokens per second).

A complex workflow agent needs capability + reliability, and can sacrifice speed (users expect to wait). Live search suggestions need speed + low cost, and can sacrifice capability (a smaller model is fine). FAQ retrieval needs cost + speed, and can use a 7B model instead of a frontier one. This is a product decision, not just an engineering one.

Inference Is OpEx

Training is CapEx — pay once, get a model. Inference is OpEx — pay continuously, for every query, every user, every session. The more successful your AI product, the higher the OpEx. This is the fundamental economic difference between AI and traditional software: in old SaaS, scale improved margins (more users on the same server). In AI SaaS, scale can compress margins (more users = more GPU cost).

Anthropic needs approximately 4–5 gigawatts of compute capacity just to serve its projected revenue. At roughly $10–13 billion per gigawatt per year in rental costs, inference economics determine whether AI labs survive or die.

First Principles Takeaway

Inference is the meter that never stops running. Every product decision you make — context length, system prompt size, conversation history policy, model selection per feature — has a direct inference cost. The product manager who understands inference economics builds features with positive unit economics. The one who doesn't builds features that become more expensive as they succeed.
