How Claude Code's Agentic Loop Works

Part 13 of the How It Actually Works series
╌╌╌╌

An engineer types a single sentence into her terminal: "Add pagination to the users API endpoint." She walks away to get coffee. When she comes back, Claude Code has read the existing codebase, identified the relevant files, written the pagination logic, updated the tests, run them, noticed a failing assertion, fixed the bug, re-run the tests, & committed the change. Twelve tool calls. Zero human intervention between the first prompt & the final result. The mechanism behind this is the agentic loop — & it's the same pattern powering every autonomous AI product being built today.

The Loop

The agentic loop is deceptively simple. Five steps, repeated: Read the user's request (or the current state of the task). Think about what to do next. Act by calling a tool — read a file, edit code, run a shell command, search the codebase. Observe the result of that action. Decide whether the task is complete or whether to loop back to step two. That's it. The model keeps cycling through think-act-observe until it determines the job is done.

What makes Claude Code's implementation interesting is the tool palette. The model can read any file in the repository, write & edit files with surgical precision, execute arbitrary shell commands, search code with grep & glob patterns, & spawn subagents — smaller, scoped instances of itself — to handle subtasks in parallel. When you ask it to refactor a module, it might spawn one subagent to understand the current architecture & another to check the test suite, then synthesize both results before making changes. The loop isn't just linear — it branches.

Context Is the Constraint

The agentic loop's biggest enemy is the context window. Every tool call adds to the conversation: the command you ran, its output, the file you read, the diff you produced. A complex task can burn through 200K tokens of context fast — esp when shell commands return verbose output or when the model reads large files. When context fills up, the model has to compact: summarizing earlier parts of the conversation to make room for new information. This is like a surgeon being told to forget the first three hours of a ten-hour operation. It works, mostly, but information loss is real.

This is why agentic coding tools behave differently from chat. In a chat interface, the human manages context — asking focused questions, providing relevant snippets. In an agentic loop, the model manages its own context, deciding what to read, what to remember, & what to compress. Good agents are parsimonious with context: they read only what they need, they search before they scan, & they keep tool outputs focused. Bad agents read entire files when they need one function, dump verbose logs into context, & run out of room before they finish.

The Permission Gradient

Autonomy without guardrails is dangerous. Claude Code implements a permission system that creates a gradient from safe to risky. Reading files? Always allowed. Writing files? Allowed in most configurations. Running shell commands? The model asks before executing anything destructive. Installing packages, deleting files, running deployment scripts — these require explicit human approval. The permission system isn't just safety theater; it's what makes the loop viable for real work. Without it, no engineer would let an autonomous agent loose in their codebase.

The design insight is that the permission boundary is configurable. Teams can pre-approve specific tools & commands, creating a "sandbox" where the agent operates freely while requiring approval for anything outside it. This lets the loop run uninterrupted for routine tasks — exactly the workflow that makes agentic coding fast — while catching the edge cases that need human judgment.

For PMs, the agentic loop is the pattern to understand. Every AI coding tool, customer support bot, research assistant, & workflow automator uses some version of read-think-act-observe. The differences between products are in the tool palette (what can the agent do?), the permission model (what is it allowed to do?), & the context management strategy (how does it remember what it's doing?). If you're building any product where AI acts autonomously, you're building an agentic loop. The question isn't whether to use this pattern — it's how well you implement each piece of it.
