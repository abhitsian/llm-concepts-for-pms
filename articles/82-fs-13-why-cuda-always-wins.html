Why CUDA Always Wins

Deep Dive 1 — From Sand to Software
┈┈┈┈

In 2006, Nvidia released CUDA — a programming framework that let developers write general-purpose code for GPUs. At the time, GPUs were for gaming. Nobody outside a few research labs cared about using them for anything else. Jensen Huang's bet was that general-purpose GPU computing would eventually matter. It took a decade to prove him right. And that decade is the moat.

The 20-Year Compound

CUDA isn't a product. It's an ecosystem. Twenty years of accumulation:

Libraries. cuBLAS for linear algebra. cuDNN for deep learning primitives. cuFFT for signal processing. TensorRT for inference optimization. NCCL for multi-GPU communication. Each library represents years of kernel-level optimization — hand-tuned assembly code that squeezes maximum performance from specific Nvidia architectures. These aren't wrappers. They're thousands of person-years of low-level engineering.

Frameworks. PyTorch — the dominant AI research framework, used in virtually every published paper since 2019 — is built on CUDA. TensorFlow. JAX (Google's framework) supports CUDA alongside its own TPU backend. Every major AI framework treats CUDA as the primary target. AMD's ROCm is a secondary backend with less testing, fewer optimizations, and more bugs.

Talent. Every ML engineer graduating from Stanford, MIT, Berkeley, or CMU learned on CUDA. Every startup's first GPU cluster runs CUDA. Every tutorial, every Stack Overflow answer, every blog post about GPU optimization assumes CUDA. The knowledge base is so deep and so CUDA-specific that switching costs aren't just technical — they're cognitive. Engineers think in CUDA patterns.

Research. When a PhD student at DeepMind invents a new attention mechanism, they implement it in CUDA first. When it gets published, other researchers reproduce it in CUDA. By the time it reaches production, it's been optimized for three generations of Nvidia hardware. The AMD or Intel version arrives 12–18 months later, if it arrives at all.

AMD's ROCm: The Challenger That Can't Close the Gap

AMD's answer to CUDA is ROCm (Radeon Open Compute). On paper, AMD's MI300X GPU matches or beats Nvidia's H100 on raw specs. More memory (192GB vs 80GB), competitive compute. AMD priced it aggressively. Major customers — Microsoft, Meta — bought MI300X clusters.

But ROCm is 8–10 years behind CUDA in software maturity. Specific problems:

Kernel coverage. CUDA has optimized kernels for virtually every operation AI models need. ROCm covers the common ones but hits gaps on newer or niche operations. When a gap appears, the researcher either writes a custom kernel (hard, slow) or switches to Nvidia (easy, fast).

Compiler quality. CUDA's compiler (nvcc) produces tighter, faster code than ROCm's compiler (hipcc) for most workloads. The difference is 10–30% on typical training jobs. That's not nothing — it's millions of dollars at scale.

Multi-GPU scaling. Nvidia's NCCL library for multi-GPU communication is battle-tested at 10,000+ GPU scale. AMD's RCCL works but has less production mileage. At frontier training scale, communication bugs that appear at 10,000 GPUs can set back a training run by weeks.

The ecosystem tax. Even if ROCm were 100% compatible today, the switching cost for a large AI lab is enormous. Revalidating every training pipeline, re-benchmarking every model, retraining staff — the total cost of switching is measured in engineering-years, not dollars.

Google's Approach: Don't Compete, Co-Design

Google looked at CUDA's moat and made a different choice: don't try to replicate the ecosystem. Instead, co-design custom hardware (TPUs) with a custom compiler (XLA) and custom framework support (JAX). The TPU doesn't run CUDA at all — it runs Google's own software stack, optimized end-to-end for Google's specific workloads.

This works for Google because they control the entire vertical: the chip, the compiler, the model architecture, the data center, the products. They don't need ecosystem breadth — they need ecosystem depth for their specific use cases. Anthropic runs much of Claude's inference on TPUs. The cost advantage is real. But this approach is only viable if you have Google's resources to build and maintain the entire stack.

Amazon's Trainium: The Cost Play

Amazon's Trainium chips take yet another angle: don't beat CUDA on performance, beat it on price. Trainium 2 offers 30–40% better price-performance than Nvidia GPUs for supported workloads. Amazon's Neuron SDK (their CUDA equivalent) is narrower but sufficient for training and inference of standard architectures.

The bet: if you can save 30–40% on your largest cost line (compute), you'll tolerate a narrower software stack. Anthropic committed to running on Trainium as part of its $4 billion Amazon partnership. The economic logic is compelling — but it means tying your compute destiny to a single cloud provider.

Why This Matters for the Next 5 Years

CUDA's moat is not permanent. It's eroding at the edges. PyTorch 2.0's compiler infrastructure (torch.compile) abstracts away some hardware specifics, making it slightly easier to target non-Nvidia hardware. Triton — an open-source GPU programming language from OpenAI — generates code for both Nvidia and AMD GPUs, reducing the kernel-writing advantage.

But "eroding at the edges" is not "collapsing." The core CUDA advantage — the libraries, the talent, the research pipeline — compounds faster than it decays. Nvidia's strategy is clear: keep CUDA just far enough ahead that switching never quite makes economic sense. Ship new hardware features that require new CUDA capabilities. Make each generation's software tightly coupled to its hardware. The treadmill never stops.

For product leaders: CUDA's dominance means Nvidia's pricing power persists. Don't plan your AI economics assuming GPU costs will fall to commodity levels quickly. The software lock-in props up hardware margins. Budget accordingly.
