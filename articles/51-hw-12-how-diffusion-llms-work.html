How Diffusion LLMs Work

Part 12 of the How It Actually Works series
╌╌╌╌

A user asks a standard LLM to write a sonnet. The model starts generating: word one, then word two, then word three — strictly left to right. By the time it reaches line 12, it realizes the rhyme scheme set up in line 1 doesn't quite work. Too bad. It already committed to those words & there's no going back. Autoregressive generation is a one-way street. Diffusion LLMs take a fundamentally different approach: start with noise, refine everything at once, & let the whole text evolve together.

Noise to Text

If you've seen how Stable Diffusion or DALL-E generates images, the core idea is familiar. Image diffusion starts with pure static — random pixel noise — & iteratively denoises it, gradually sharpening shapes, colors, & details until a coherent image emerges. Diffusion LLMs apply the same principle to text. Start with a sequence of random or masked tokens — pure noise in token space. Then run multiple denoising passes, each one making the sequence slightly more coherent. After enough steps, the noise resolves into fluent, meaningful text.

The key difference from autoregressive models is parallelism. A standard LLM must generate token N before it can generate token N+1, cuz each token depends on all previous ones. Diffusion LLMs refine all positions simultaneously. In each denoising step, the model looks at the current (noisy) state of the entire sequence & pushes every token toward a better version. This means the model can, in principle, generate an entire paragraph in a fixed number of denoising steps rather than one token at a time.

The Revision Advantage

Parallel generation is the speed story, but the deeper advantage is revision. Autoregressive models are trapped by their own left-to-right commitment. Once token 5 is generated, tokens 6 through 500 must accommodate it — even if token 5 was a suboptimal choice that makes the rest worse. Diffusion LLMs don't have this constraint. Early denoising steps establish rough structure — the general topic, approximate sentence boundaries, key concepts. Later steps refine word choice, fix coherence issues, & tighten the prose. Token 5 can change based on what token 50 looks like, cuz both are being refined simultaneously.

This mirrors how ppl actually write. Nobody writes a polished paragraph left to right in a single pass. You draft, revise, rearrange, reconsider. Diffusion LLMs bake this revision process into the generation mechanism itself. For tasks where global coherence matters — poetry, structured arguments, code with interdependent functions — the ability to revise earlier tokens based on later context is a genuine architectural advantage.

Where It Stands Today

Inception Labs' Mercury model demonstrated the practical potential: 10x+ speed improvements on certain benchmarks compared to autoregressive models of similar size. The model generates text fast enough for real-time applications that were previously impractical — live transcription with simultaneous editing, instant code completion for entire functions, rapid structured output generation. The speed comes from needing only a fixed number of denoising steps (typically 10-50) regardless of output length, compared to autoregressive models that need one forward pass per token.

But diffusion LLMs are still early. The quality on open-ended generation tasks — the kind of creative, conversational, nuanced writing that frontier autoregressive models handle well — isn't quite there yet. Training recipes are less mature. The ecosystem of fine-tuning tools, RLHF pipelines, & deployment infrastructure that's been built around autoregressive models over the past few years doesn't transfer directly. Mercury showed the architecture works; it hasn't yet shown it wins across the board.

For PMs, diffusion LLMs are worth tracking esp if your product is latency-sensitive. Chat interfaces, real-time collaboration tools, coding assistants that need to generate entire blocks instantly — these are the use cases where the speed advantage matters most. The quality gap will likely narrow as research matures, & when it does, the speed-quality tradeoff changes fundamentally. Products that were designed around the constraint of slow, sequential generation may find that constraint suddenly lifted. The PMs who planned for that possibility will ship faster than the ones who have to redesign.
